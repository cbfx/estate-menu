# vinext → Cloudflare Workers scaffold with deploy-on-merge CI

**Date:** 2026-06-20
**Status:** Approved, pending implementation plan

## Goal

Scaffold a [vinext](https://github.com/cloudflare/vinext) app (Cloudflare's Vite-based
Next.js reimplementation) that deploys to **Cloudflare Workers** via GitHub Actions on
merge to `main`, using a single GitHub **production** Environment for secrets.

The app itself is a bare default starter — the real "estate-menu" app is designed later.
This work delivers only the scaffold + Cloudflare config + CI pipeline.

**Definition of done (the clever part):** the pipeline is wired so that `git init` → push →
CI **runs and attempts to deploy, then fails at the Cloudflare auth step** because no
secrets exist yet. That failure is the proof the pipeline is correctly assembled.

## Decisions (confirmed with user)

| Decision | Choice |
|---|---|
| Router | **App Router** (RSC) — needs `@vitejs/plugin-rsc` + `react-server-dom-webpack` |
| Package manager | **pnpm** (`pnpm-lock.yaml`) |
| CI deploy method | **Commit config + `wrangler deploy`** via `cloudflare/wrangler-action` (config reviewable in repo, not generated at deploy time) |
| Node version | 22 |
| Worker name | `estate-menu` |
| Account ID | Supplied via secret `CLOUDFLARE_ACCOUNT_ID`, **not** committed to `wrangler.jsonc` |
| CI scope | One workflow: deploy-on-push-to-main only. No PR checks, no preview envs. |

## Components

### App
Default Next.js App Router starter, migrated to vinext: a single minimal placeholder page.
Intentionally bare. The user designs the real app afterward.

### Local Cloudflare config
`vite.config.ts`, `wrangler.jsonc`, and `worker/index.ts` are written and committed directly
(rather than auto-generated by `vinext deploy` at deploy time), so the config is reviewable
and the CI step is a plain `wrangler deploy`. `wrangler.jsonc` carries the worker name, assets
binding, and a compatibility date, but **no account ID** — that arrives via the
`CLOUDFLARE_ACCOUNT_ID` secret/env, keeping the repo free of account specifics.

### CI — `.github/workflows/deploy.yml`
- **Trigger:** `push` to `main` only (a merge to main is a push). Single job.
- **Job is scoped to `environment: production`** → binds to the GitHub production Environment
  so its secrets and protection rules apply.
- **Steps:** checkout → setup pnpm + Node 22 → `pnpm install --frozen-lockfile` →
  `pnpm build` (`vinext build`) → `cloudflare/wrangler-action` running `wrangler deploy`.
- **Secrets consumed (from the production environment):** `CLOUDFLARE_API_TOKEN`,
  `CLOUDFLARE_ACCOUNT_ID`. Neither exists at handoff → the deploy step fails at auth.

## Data flow

1. Developer merges PR to `main` (or pushes to `main`).
2. GitHub Actions triggers `deploy.yml`, job enters the `production` environment.
3. Build runs (`vinext build`) producing Worker + assets output.
4. `wrangler deploy` reads `wrangler.jsonc` + `CLOUDFLARE_API_TOKEN`/`CLOUDFLARE_ACCOUNT_ID`.
5. With no secrets set → **fail at auth** (expected end state).
   With secrets set later → deploy to Cloudflare Workers (green).

## Error handling / expected failure

The missing-credentials failure is intended and is the acceptance signal. The README documents
the two secrets to add to the production environment to turn the pipeline green:
- `CLOUDFLARE_API_TOKEN` (Cloudflare dashboard → "Edit Cloudflare Workers" token template)
- `CLOUDFLARE_ACCOUNT_ID`

## Verification

What is verified during implementation (in this folder):
- `pnpm install` succeeds; lockfile generated.
- `pnpm dev` (`vinext dev`) serves the default page locally. **This is the primary gate** for
  confirming the dep list and `vite.config.ts` are correct — trusted over memory, since vinext
  is experimental.
- `pnpm build` (`vinext build`) produces deployable output.
- `wrangler deploy --dry-run` validates `wrangler.jsonc` with no credentials.
- `deploy.yml` is valid YAML, references `environment: production`, and uses the exact secret
  names `CLOUDFLARE_API_TOKEN` / `CLOUDFLARE_ACCOUNT_ID`.

What is the user's final manual check (cannot be done here — no GitHub remote):
- create GitHub repo → push → watch Actions run and fail at the Cloudflare auth step.

## Risks / notes

- **vinext is experimental.** The exact dependency list and the contents of `vite.config.ts`
  and `worker/index.ts` will be pinned against the live `cloudflare/vinext` repo at
  implementation time, using "`vinext dev` serves the page" as the verification gate rather
  than relying on remembered templates.
- Whether `next` remains an installed dependency (for `next/link`, `next/navigation`, types)
  vs. being shimmed by vinext will be confirmed against the repo during implementation.

## Out of scope (YAGNI)

- PR build/lint/typecheck checks.
- Preview / staging environments.
- Custom domains, KV/D1/R2 bindings, ISR cache tuning.
- The actual estate-menu application features.
